## Findings by Category

### Flaws — 6.5/10

1. **critical** `src/scanner.js:420` (`scanRecursive`)
   Batch project scans use Promise.all instead of Promise.allSettled. If any single scan() call throws (e.g., ConfigParseError from loadConfig, or I/O failure in loadChecks), Promise.all rejects immediately and every other project in that 4-slot batch is silently abandoned. runChecks() correctly uses Promise.allSettled for individual checks (line 194), but the outer batch wrapper does not.
   **Fix:** Replace `Promise.all(batch.map(...))` at line 420 with `Promise.allSettled(batch.map(...))` and handle rejected outcomes individually, similar to runChecks(). Failed projects should surface as a score-0 entry with a critical finding, not crash the whole recursive run.

2. **warning** `src/checks/mcp-config.js:433` (`run`)
   KEY_PATTERNS are tested with pattern.test(fullCommand) without resetting pattern.lastIndex = 0 first. utils.js scanLineForSecrets() includes an explicit defensive reset (line 385) with a comment explaining the /g-flag hazard. The same KEY_PATTERNS array is used here without that guard, creating an inconsistency: if any pattern ever acquires the /g flag, mcp-config.js would produce intermittent false negatives while scanLineForSecrets() would remain protected.
   **Fix:** Add `pattern.lastIndex = 0;` before each `pattern.test()` call in the inline-credentials loop, or extract a shared helper that includes the defensive reset.

3. **warning** `src/checks/mcp-config.js:510` (`run`)
   Inside `if (packageName && context.online)`, `const registryResult = await checkNpmRegistry(packageName)` declares a block-scoped variable named registryResult that shadows the outer `let registryResult` (line 221, the MCP registry fetch result). Two completely unrelated objects share the same name within the same function. If the inner block ever expands or is refactored, it is easy to accidentally read the npm check result where the MCP registry result is intended, or vice versa.
   **Fix:** Rename the inner variable: `const npmCheckResult = await checkNpmRegistry(packageName)`.

4. **warning** `src/checks/index.js:22` (`loadChecks`)
   Module-level `_registeredFixes = {}` is reset AFTER the `await fs.promises.readdir` yield point (line 16), so concurrent loadChecks() calls — which scanRecursive() triggers via Promise.all with CONCURRENCY=4 — can interleave their reset and population phases. Scan-1 resets the object, starts populating; Scan-2 resets it to {} again mid-population. While --fix is currently single-scan only (so getRegisteredFixes() is never called concurrently), this is a latent correctness bug for any future recursive-fix mode.
   **Fix:** Move the `_registeredFixes = {}` reset to before the `await` on line 16, or switch to returning the fixes map as a function return value rather than storing in module-level state.

5. **warning** `src/scoring.js:20` (`calculateCheckScore`)
   Lines 20-21 compare deduction against hardcoded literals `-15` and `-2`, and line 24 multiplies by hardcoded `15` and `2`. SEVERITY_DEDUCTIONS in constants.js is declared as the single source of truth for these values. If the deduction magnitudes ever change, calculateCheckScore silently miscounts warnings and info findings while still compiling cleanly.
   **Fix:** Replace the hardcoded comparisons and multipliers with references to SEVERITY_DEDUCTIONS: `if (deduction === SEVERITY_DEDUCTIONS[SEVERITY.WARNING]) warningCount++; if (deduction === SEVERITY_DEDUCTIONS[SEVERITY.INFO]) infoCount++;`, and compute the score using the constant values rather than duplicated literals.

6. **info** `src/index.js:349` (`run`)
   The condition in `process.exit(added.length > 0 ? 1 : 0)` is always true at that point: the `added.length === 0` branch exits at line 336 unconditionally. The ternary dead-codes the `0` branch and misleads readers into thinking this location handles the empty-added case.
   **Fix:** Simplify to `process.exit(1)` and add a brief comment referencing the guard at line 334.

7. **info** `src/checks/index.js:16` (`loadChecks`)
   fs.promises.readdir(__dirname) returns files in OS-defined order, which is non-deterministic on some filesystems (e.g., HFS+, some ext4 configurations). Check load order determines which check wins tie-breaking in deduplicateFindings when two checks have equal weight. No weight ties exist today (all non-zero weights in WEIGHTS are unique), but the ordering guarantee is implicit rather than enforced.
   **Fix:** Sort checkFiles after filtering: `const checkFiles = files.filter(...).sort();` to guarantee alphabetical and reproducible check load order.
