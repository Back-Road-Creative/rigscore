### Flaws — 7/10

1. **warning** `src/scanner.js:39` (`deduplicateFindings`)
   Incomplete cross-check deduplication for per-file findings. When a check has multiple findings with the same normalized title (all skipped by the `prev.resultIdx === ri` continue), `seen` stores only ONE entry — the first one visited in backward traversal (highest index). When a higher-weight check later supersedes, only that single stored entry is added to `toRemove`. All other per-file variants with the same normalized title from the losing check survive, inflating finding counts whenever two checks overlap on a per-file basis.
   **Fix:** After the cross-check supersede branch (lines 41-44), scan the losing result's findings for all remaining entries sharing the same key and add them to `toRemove` as well, or change the within-check guard to record all per-file indices into a list instead of overwriting a single entry.

2. **warning** `src/index.js:251` (`run`)
   Non-recursive scan path swallows ALL exceptions with `process.exit(2)` and a generic message (lines 254–257), while the recursive path calls `handleFatal(err)` which only catches `ConfigParseError` and rethrows everything else (line 215: `throw err`). Unexpected Node.js runtime errors or check bugs in the dominant single-scan invocation produce an uninformative 'Error: scan failed: <message>' and suppress stack traces, making them hard to debug.
   **Fix:** Replace the inline try/catch in the non-recursive path with `handleFatal(err)` (already defined at line 210), adding a final `process.exit(2)` only for truly unhandled errors to match the recursive path's re-throw behavior.

3. **warning** `src/checks/mcp-config.js:510` (`run`)
   Inside the server loop, `const registryResult = await checkNpmRegistry(packageName)` shadows the outer `let registryResult` (set at line 224 from `fetchRegistry()`). The two have completely different shapes: outer is `{ servers: Array, warning?: string }`, inner is a finding object `{ findingId, severity, title, … }` or `null`. Block scoping prevents a runtime failure today, but any future code added between line 510 and the outer loop exit that reads `registryResult` expecting the MCP registry object will silently use the npm-registry finding object instead.
   **Fix:** Rename the inner variable to `npmFinding` (or similar) to remove the shadowing.

4. **warning** `src/checks/index.js:62` (`discoverPlugins`)

This causes the check to run twice, doubling its findings and inflating/distorting the score.

### Security — 8.5/10

1. **warning** `src/checks/mcp-config.js:119` (`checkNpmRegistry`)
   npm registry response is accumulated with no size cap (`let data = ''; res.on('data', chunk => data += chunk)`). A popular or adversarially large package metadata document (some exceed 10 MB) can exhaust process memory. This path is only reachable when the user passes --online, but the scanned MCP config controls which package name is fetched.
   **Fix:** Add a byte counter and abort the request once a threshold (e.g. 512 KB) is exceeded: if (data.length > MAX_BYTES) { req.destroy(); resolve(null); return; }.

2. **uncertain** `src/checks/mcp-config.js:378`
   Comment at line 378 labels the check 'CVE-2026-21852' but the learnMore URL at line 387 references a post about 'CVE-2025-59536'. If the CVE identifier exposed in the finding is wrong, operators following it for remediation guidance will land on a different vulnerability. Cannot verify correctness from local files alone.
   **Fix:** Audit and reconcile the CVE identifiers; reference only the CVE that matches the ANTHROPIC_BASE_URL redirect class of vulnerability in both the comment and the learnMore URL.

3. **info** `src/checks/deep-secrets.js:158`
   Finding detail exposes the first 30 characters of the detection regex source: `detail: \`Pattern: ${result.pattern.source.slice(0, 30)}...\``. JSON output consumers (CI pipelines, SARIF uploads) receive this, effectively documenting which regex patterns to craft around.
   **Fix:** Replace with a stable, human-readable pattern label (e.g. the pattern name/tag) instead of the raw regex source.

4. **info** `src/mcp-registry.js:37` (`getDefaultCachePath`)
   `process.env.XDG_CACHE_HOME` is consumed unsanitized as the base directory for the registry cache file. An attacker who controls the process environment (e.g. via a malicious pre-commit hook that sets XDG_CACHE_HOME before invoking rigscore) could redirect the cache write to an arbitrary path such as /etc/cron.d/rigscore.
   **Fix:** Validate that the resolved cache path stays within a safe prefix, or document that XDG_CACHE_HOME is operator-trusted. At minimum, ensure mkdir is not called with a root-owned path by checking that the resolved path is under homedir.

5. **info** `src/http.js:22` (`fetchHeaders`)
   Protocol selection uses `url.startsWith('https')` rather than a strict `url.startsWith('https://')` check. A URL like `httpsfoo://...` would route through the `https` Node module, likely failing silently. Not exploitable in the current call sites (controlled URL construction), but fragile if new callers are added.
   **Fix:** Use `url.startsWith('https://')` (include the slashes) for protocol dispatch.
