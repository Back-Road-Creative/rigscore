### Security — 7.5/10

1. **warning** `src/checks/index.js:108` (`discoverPlugins`)
   Plugin discovery calls `import(pluginPath)` for any package named `rigscore-check-*` found in the *scanned project's* node_modules — not just rigscore's own install. An attacker who controls the target directory can plant a malicious `rigscore-check-evil` package and get arbitrary code executed inside the scanner's process whenever rigscore scans that directory.
   **Fix:** Restrict plugin search to rigscore's own install tree only (remove the `cwd`-relative search path), or sandbox plugin execution. At minimum, emit a prominent warning when a plugin is loaded from the target project's node_modules and require an explicit allowlist opt-in via config.

2. **warning** `src/checks/site-security.js:322` (`run`)
   URLs from `context.config.sites` (read from the scanned project's `.rigscorerc.json`) are passed directly to `fetchHeaders()`, `fetchBody()`, and `probeStatus()` with no scheme allowlist or host blocklist. When rigscore is run with `--online` against a malicious project, a crafted `.rigscorerc.json` with `"sites": ["http://169.254.169.254/latest/meta-data/"]` triggers SSRF requests to cloud metadata APIs or internal network services.
   **Fix:** Before fetching, validate that each URL uses the `https:` or `http:` scheme and reject or warn on private/loopback IP ranges (127.x, 10.x, 172.16-31.x, 192.168.x, 169.254.x). Apply the same guard in `checkExposedPaths`.

3. **info** `src/http.js:40` (`fetchBody`)
   `fetchBody()` appends all response chunks to an in-memory array with no size cap. A malicious site URL that streams a multi-GB response can exhaust the scanner process's memory.
   **Fix:** Add a running byte counter inside the `data` handler and call `res.destroy()` once a configurable cap (e.g. 10 MB) is exceeded, then resolve with null or the truncated partial body.

4. **info** `src/mcp-registry.js:37` (`getDefaultCachePath`)
   `XDG_CACHE_HOME` is consumed directly from the environment without sanitization: `const xdg = process.env.XDG_CACHE_HOME && process.env.XDG_CACHE_HOME.trim()`. A compromised environment with `XDG_CACHE_HOME=../../some/sensitive/path` causes cache writes (`saveRegistryCache`) to an unintended filesystem location.
   **Fix:** Resolve the path to an absolute form and verify it is under the user's home directory before using it, or document that XDG_CACHE_HOME is trusted and the attack requires local shell access.

5. **info** `.gitignore`
   The `.gitignore` contains no patterns for `.env`, `*.pem`, or `*.key` files. The project currently has no such files, but the omission means future additions (e.g. a `.env` with test credentials) would be silently committed.
   **Fix:** Add `.env`, `.env.*`, `*.pem`, `*.key`, and `*.p12` to `.gitignore` as a hygiene baseline.

6. **info** `src/checks/mcp-config.js:118` (`checkNpmRegistry`)
   The `checkNpmRegistry` function accumulates the full npm registry JSON response in a string (`data += chunk`) with no size cap. A non-standard or proxied registry could serve an arbitrarily large payload and exhaust memory.
   **Fix:** Enforce a response size cap (e.g. 1 MB) and abort the request if the threshold is exceeded, consistent with the fix recommended for `fetchBody`.

### Production — 7.5/10

1. **warning** `bin/rigscore.js:136`
   `run(args)` is called without `await` and without `.catch()`. Because `run` is an `async` function, any error that propagates out of it (including the `throw err` re-throw in `handleFatal` at src/index.js:215) becomes an unhandled promise rejection. Node.js 15+ terminates the process with exit code 1 and prints a raw stack trace — users see engine internals instead of a clean error message.
   **Fix:** Change line 136 to `await run(args)` and wrap in a top-level `try/catch` that writes a user-friendly message to stderr before calling `process.exit(2)`. Alternatively, append `.catch(err => { process.stderr.write(`Error: ${err.message}\n`); process.exit(2); })` to the call.

2. **warning** `bin/rigscore.js`
   No `process.on('unhandledRejection', ...)` handler is registered anywhere in the codebase. Any async bug in a check module or scanner path that throws and is not explicitly caught will produce a bare Node.js stack trace for end users — a poor CLI experience and a potential source of leaking internal file paths.
   **Fix:** Add `process.on('unhandledRejection', (err) => { process.stderr.write(`rigscore: unexpected error: ${err?.message ?? err}\n`); process.exit(2); });` in `bin/rigscore.js` before dispatching to subcommands or calling `run()`.

3. **warning** `.github/workflows/docker-publish.yml:61`
   The `docker/build-push-action` step does not pass `--build-arg VERSION=<tag>` to the build. The Dockerfile declares `ARG VERSION=dev` (line 5) and stamps the OCI label `org.opencontainers.image.version` from it — so every published image, even release tags, carries the version label `dev`. Provenance and registry tooling that reads OCI labels will see incorrect version information.
   **Fix:** Add `build-args: VERSION=${{ github.ref_name }}` (strip the `v` prefix if needed) to the `docker/build-push-action` step inputs in docker-publish.yml.

4. **warning** `.github/workflows/release.yml:88`
   The `Generate headlessmode release post` step depends on `secrets.HEADLESSMODE_PAT`. If the secret is absent or expired the step fails with a non-zero exit (`set -euo pipefail`) — but the GitHub release was already created in the previous step, leaving a release with no accompanying blog post and no clear failure signal to the caller. The workflow has no `continue-on-error` guard nor a notification step.
   **Fix:** Either add `continue-on-error: true` plus a Slack/GitHub-issue notification on failure, or add a pre-flight secret check (`if: secrets.HEADLESSMODE_PAT != ''`) at the job level and surface a clear warning when it's missing.

5. **info** `.github/workflows/ci.yml:35`
   CI runs tests but has no coverage threshold step. The workspace CLAUDE.md documents a 40% coverage floor for the workspace CI, but pkg-rigscore's own CI does not enforce coverage at all. Regressions in check coverage can slip through undetected.
   **Fix:** Add `--coverage --coverage.thresholds.lines 40` to the `npm test` invocation in ci.yml, or add a dedicated `run: npx vitest run --coverage` step after the test step with a `coverageThreshold` configured in vitest config.

6. **info** `src/index.js:255` (`run`)
   The generic scan-error handler writes `err.message` verbatim to stderr (`Error: scan failed: ${err.message}`). For a security-scanner CLI, error messages from check modules could include absolute file paths or partial secret values from context passed through exception chains — mild information-leakage risk.
   **Fix:** Sanitize `err.message` before writing it, or catch check-level errors closer to the source and emit structured `findingId`-linked messages instead of raw exception text.

7. **info** `package.json`
   Runtime dependencies (`chalk`, `yaml`) are pinned to exact patch versions, which is good. However there is no `npm audit` step in CI, so newly-disclosed CVEs in those dependencies would not be caught until someone runs it manually.
   **Fix:** Add `- run: npm audit --audit-level=moderate` as a CI step, or enable GitHub's Dependabot alerts for the repository.
