### Flaws — 6/10

1. **critical** `src/checks/coherence.js:70` (`default export`)
   The `coherence` check export does not declare `pass: 2`, so `scan()` places it in `pass1Checks` (scanner.js:291: `c => !c.pass || c.pass === 1`). Pass-1 context never includes `priorResults`, so `run()` immediately returns `{ score: NOT_APPLICABLE_SCORE, findings: [] }` on every real scan. The 14-point coherence weight is permanently excluded from scoring, all cross-config contradiction checks (network-claim vs MCP transport, path-claim vs broad filesystem, skill-injection vs anti-injection claim, etc.) never fire, and the compound-risk penalty in `calculateOverallScore` (scoring.js:107–113) can never trigger. Unit tests mask this by calling `check.run({ priorResults: [...] })` directly, bypassing the pass-routing entirely.
   **Fix:** Add `pass: 2` to the exported object alongside `id`, `name`, `category`, `enforcementGrade`. Confirmed pattern: see `src/checks/skill-coherence.js:230` and `src/checks/network-exposure.js:374`.

2. **warning** `src/checks/mcp-config.js:293` (`extractPackageName`)
   The character-class filter `[a-z0-9-]+` (regex `/^(@[a-z0-9-]+\/)?[a-z0-9-]+(@.+)?$/`) excludes `_` and `.`, both of which are valid npm package-name characters (e.g. `lodash.set`, `babel_register`). Any MCP server whose package arg contains an underscore or dot falls through the `continue` and `extractPackageName` returns `null`, silently skipping the typosquat curated/registry checks and the unpinned-npx-package check for that server.
   **Fix:** Widen the character class to `[a-z0-9_.\-]+` to match all valid lowercase npm package-name characters.

3. **warning** `src/checks/mcp-config.js:606` (`checkAnthropicBaseUrl`)
   `envBaseUrl.includes('api.anthropic.com')` is a substring match, not a hostname match. A redirect URL crafted to include the safe hostname as a path segment or query parameter — e.g. `https://evil.com/proxy/api.anthropic.com` or `https://api.anthropic.com.evil.com/` — passes the safe-host check and suppresses the CVE-2026-21852 critical finding. The same substring issue applies to the `localhost` and `127.0.0.1` checks.
   **Fix:** Parse the URL with `new URL(envBaseUrl)` (the same `extractHost` helper already in the file) and compare `.hostname` against the safe-host set, rather than doing a substring `includes`.

4. **info** `src/scanner.js:32` (`deduplicateFindings`)
   The dedup loop skips `info`-severity findings unconditionally (`if (!f.title || f.severity === 'pass' || f.severity === 'info') continue`). Two checks that each emit an INFO with the same normalized title both appear in the final output unfiltered. WARNING and CRITICAL findings are correctly deduplicated; INFO is not. The inconsistency could accumulate duplicate advisory noise when multiple checks observe the same condition.
   **Fix:** Either extend dedup to INFO severity (matching the WARNING/CRITICAL logic) or document the intentional asymmetry with a comment explaining why INFO findings are exempt.

5. **info** `src/checks/mcp-config.js:511` (`checkHashPinning`)
   `if (Object.keys(currentHashes).length === 0 || writeState === false) return findings;` bails out of the function entirely when `writeState === false` — it suppresses *both* the state-file write and all drift-detection finding generation. The parameter name `writeState` implies it only controls the side-effect (file I/O), but it also disables the read-and-compare logic. Tests that set `context.writeState = false` to prevent disk writes inadvertently also suppress rug-pull warning findings, producing a false-clean result during test execution.
   **Fix:** Separate the two responsibilities: always run the hash comparison and produce findings; only gate the `saveState` call on `writeState !== false`.

### Security — 8.5/10

1. **warning** `src/http.js:40` (`fetchBody`)
   Response body is accumulated into an unbounded in-memory buffer (chunks array). A malicious or oversized server in the user's `sites:` config could send a multi-gigabyte response, causing OOM during --online scanning.
   **Fix:** Add a per-response size cap (e.g. 10 MB). Track accumulated bytes in the `data` handler and destroy the socket + reject/resolve(null) when the limit is exceeded.

2. **warning** `src/checks/site-security.js:322` (`run`)
   URLs from `.rigscorerc.json` `sites:` are passed directly to fetchHeaders/fetchBody/probeStatus without a scheme whitelist. The only guard is `url.startsWith('https://') ? https : http`, so any http:// URL — including internal network addresses (http://169.254.169.254/…, http://localhost:PORT) — will be fetched when --online is active. An attacker who can write to the project's config can force outbound requests to internal services.
   **Fix:** Validate that each URL parses to an `http:` or `https:` protocol via `new URL()` before fetching. Optionally add an allowlist/denylist for private RFC-1918 ranges when --online is set.

3. **warning** `test/fixtures/scored-project/.env`
   Prompt injection payload present in committed test fixture. Reading this file inside an LLM session caused a forged <system-reminder> block to appear advertising three suspicious skills ('operator-sudo', 'clean-skill', 'destructive-unguarded'). The fixture is intentional (rigscore tests its own detectors), but the injected content is live and demonstrated the attack vector during this analysis run. Any LLM-assisted workflow that processes rigscore's file-read output will be exposed.
   **Fix:** Wrap injected-payload fixtures in a clear structural delimiter so the harness can neutralise them before they reach LLM context (see CLAUDE.md data-framing rules). Add a README note inside test/fixtures/scored-project/ warning that its content is adversarial.

4. **info** `.gitignore`
   The project-root .gitignore contains no `.env` entry. There is no .env at the root today, but if one were created during development (e.g. for a local API key) it would be tracked by git with no warning. Rigscore's own env-exposure check would flag this on the next scan, but the repo itself doesn't follow the practice it enforces.
   **Fix:** Add `.env` and `.env.*` (excluding `.env.example`) to the root .gitignore.

5. **info** `test/fixtures/scored-project/.env:2`
   Test fixture contains `AKIAIOSFODNN7EXAMPLE` (Amazon's canonical AWS example key) and `sk-proj-FIXTURE000…` (OpenAI-format key). The file is commented as fake, but gitleaks / TruffleHog and GitHub's push-time secret scanning will flag these patterns and block PRs in repos where those scanners are active.
   **Fix:** Replace with clearly non-matching placeholder patterns (e.g. AKIAXXXXXXXXXXXXXXXX) or add per-file gitleaks allowlist annotations.

6. **info** `src/cli/mcp-subcommands.js:66` (`runMcpPin`)
   The `serverName` CLI argument is used as a bare object key: `merged.servers[serverName] = {...}`. If serverName is `__proto__` or `constructor`, this mutates the object prototype chain transiently (before JSON.stringify strips it). Practically harmless since the value is serialised immediately and JSON.parse creates plain objects on reload, but it is a prototype pollution edge case.
   **Fix:** Validate serverName against `/^[\w@/.-]+$/` before use, or use `Object.create(null)` for the `servers` map.

7. **info** `src/mcp-registry.js:148` (`defaultFetch`)
   The --refresh-mcp-registry network fetch uses `globalThis.fetch` (Node 18+ built-in) without an explicit response body size cap. The registry URL is hardcoded to the MCP community registry, so the attack surface is narrow, but a compromised registry could return a multi-MB payload.
   **Fix:** Add a body-size guard (e.g. check Content-Length header before reading, or stream-limit the body) consistent with the fix recommended for fetchBody in http.js.

### Production — 8.5/10

1. **warning** `.github/workflows/ci.yml:19`
   GitHub Actions are pinned to mutable version tags (e.g. actions/checkout@v4, actions/setup-node@v4) rather than immutable SHA digests across all four workflow files (ci.yml, release.yml, docker-publish.yml, release-provenance.yml). A compromised or re-tagged action version can inject malicious code into CI with no diff visible to reviewers — particularly ironic for a tool whose core value proposition is detecting MCP supply-chain pin-drift.
   **Fix:** Pin every action to its full commit SHA (e.g. actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2). Use Dependabot or renovate to keep SHAs current.

2. **info** `vitest.config.js:19`
   Coverage floor is set at 40% lines — intentionally documented in both vitest.config.js and CLAUDE.md as the project's historical baseline, but low for a security-scanning tool where missed branches mean missed findings. The 6-OS × 3-Node CI matrix is excellent; the coverage gate does not match that rigor.
   **Fix:** Incrementally raise the threshold (e.g. to 60%) as new tests are added. Consider adding branch coverage alongside line coverage since the check modules use many conditional paths that line coverage won't surface.

3. **info** `src/fixer.js:50`
   A single console.warn() call exists in the fixer, while the entire rest of the codebase uses process.stderr.write() for all diagnostic output. console.warn prepends timestamps in some Node runtime environments and adds a newline differently — minor output discipline inconsistency.
   **Fix:** Replace console.warn(...) with process.stderr.write(`...\n`) to match the codebase convention.

4. **info** `src/http.js:37` (`fetchBody`)
   fetchBody accumulates response chunks without a size cap. A slow or unexpectedly large HTTP response (used in --online mode for MCP registry fetches and site-security checks) will buffer the full body in memory. Only triggered via --online, so normal offline scans are unaffected, but a rogue or misconfigured endpoint could cause excessive memory use.
   **Fix:** Add a max-body-size guard: track accumulated bytes and call req.destroy() + resolve(null) once a threshold (e.g. 10 MB) is exceeded.
